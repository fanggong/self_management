# 总体技术架构规范

## 目的
- 定义系统的模块边界、调用方向、数据流方向和变更落点。
- 为新线程提供统一的架构判断标准，避免将职责写错层、调错服务或混淆数据边界。

## 适用范围
- 适用于所有涉及 `frontend`、`admin-api`、`sync-worker`、`dbt`、数据库、任务调度和连接器的任务。
- 适用于功能新增、问题修复、架构调整、数据链路扩展和跨服务联调。

## 规范正文

### 模块划分
- 系统必须划分为以下核心模块：
  - `frontend`
    负责用户界面、导航、交互、表单输入、可视化展示和前端态管理。
  - `admin-api`
    负责统一鉴权、业务 API、应用事务数据处理、跨服务编排和对外契约稳定性。
  - `sync-worker`
    负责连接器验证、同步调度、原始数据抓取、同步任务执行和运行状态回写。
  - `dbt-runner`
    负责 dbt 模型目录发现、模型运行、运行日志产出和模型产物读取。
  - `PostgreSQL`
    负责应用事务数据、原始接入数据和 dbt 产出数据的持久化。
  - `Redis`
    负责 Celery broker、结果后端和任务调度辅助，不得承载长期业务真相。

### 调用方向
- `frontend` 只能调用 `admin-api`，不得直接调用 `sync-worker`、`dbt-runner` 或数据库。
- `admin-api` 可以调用：
  - `sync-worker` 的内部能力，用于连接器验证、报告解析或内部协作。
  - `dbt-runner` 的内部能力，用于模型清单查询、模型执行和日志读取。
- `sync-worker` 可以调用：
  - `admin-api` 的内部能力，用于触发平台级调度或回调必要的内部业务入口。
  - `PostgreSQL` 与 `Redis`
  - 必要的外部连接器源站
- `dbt-runner` 只能调用：
  - 本地 dbt 命令
  - `PostgreSQL`
- 任何浏览器端能力都禁止直接依赖 `sync-worker` 或 `dbt-runner` 的内部接口。

### 职责边界
- `frontend` 必须只负责展示、交互、轻量校验和前端状态，不得承载业务编排真相。
- `admin-api` 必须是对外业务协议的唯一稳定入口。
- `sync-worker` 必须只负责连接器相关的执行与原始落库，不负责面向用户界面的展示逻辑。
- `dbt-runner` 必须只负责模型运行与元数据读取，不负责业务鉴权和应用事务。
- 数据转换职责必须优先落在 dbt，而不是散落在 `admin-api` 或 `frontend`。

### 数据分层
- 数据库必须按职责分层，至少包含以下逻辑层：
  - `app`
    保存账户、用户、连接器配置、同步任务、运行历史等事务型数据。
  - `raw`
    保存外部来源的原始载荷及采集血缘。
  - `staging`
    将 `raw` 数据结构化为可查询表，并保留必要血缘字段。
  - `intermediate`
    对 `staging` 做语义整理、清洗和统一。
  - `marts`
    向看板、查询和业务消费提供稳定模型。
- 新增字段或模型时，必须先判断其所属层，不得跨层硬塞。

### 数据流规范
- 连接器数据流必须遵循：
  `外部数据源 -> sync-worker -> raw -> dbt staging -> dbt intermediate -> dbt marts -> admin-api -> frontend`
- 用户配置流必须遵循：
  `frontend -> admin-api -> app`
- dbt 执行流必须遵循：
  `frontend -> admin-api -> dbt-runner -> dbt -> PostgreSQL`
- 连接器验证流必须遵循：
  `frontend -> admin-api -> sync-worker -> 外部数据源`

### 配置分层
- 所有运行配置必须通过环境变量或运行时配置注入，不得将密钥硬编码在业务代码中。
- 面向浏览器暴露的配置必须与服务内部配置分离。
- 连接器凭证必须加密存储，明文只允许出现在提交时、验证时或运行时内存中。
- 任务调度、时区、内部服务地址和数据库连接信息必须由部署环境统一注入。

### 调度与执行
- 定时任务职责必须归属于调度器与 `sync-worker`，不得由 `frontend` 或普通用户请求直接承担周期执行职责。
- 同步任务必须有显式状态流转，至少包括：
  `queued -> running -> success/failed`
- 同步成功的判定必须以原始采集链路完成为准，不得把下游所有消费步骤隐式捆绑为同一成功条件，除非文档另有明确规定。

### 变更落点规则
- 变更页面表现或交互，必须优先落在 `frontend`。
- 变更对外接口、鉴权语义或事务业务，必须优先落在 `admin-api`。
- 变更连接器抓取、验证、任务调度或原始落库，必须优先落在 `sync-worker`。
- 变更数据解析、语义整理或消费模型，必须优先落在 `dbt`。
- 变更跨层能力时，必须先明确主责任层，再让其他层做最小配套。

## 边界与例外
- 连接器中的人工解析确认型流程允许在 `admin-api` 中完成部分业务编排，但原始落库规范不得绕过 `raw` 层。
- `dbt-runner` 可以暴露内部接口给 `admin-api`，但不得被视为公共业务服务。
- 未在本规范中明确授权的跨服务直连一律视为禁止。

## 实施清单
- 变更前先确定主责任模块。
- 变更前先判断数据属于 `app/raw/staging/intermediate/marts` 哪一层。
- 新增服务调用前先确认是否符合允许的调用方向。
- 新增任务流程前先定义状态流转和回写位置。
- 新增配置前先判断是浏览器公开配置还是服务内部配置。
