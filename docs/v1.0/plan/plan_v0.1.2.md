# plan_v0.1.2 修订版（前端渐进式执行）

## 简述
本计划用于交付给研发 agent，作为 v0.1.2 前端开发的执行基线。  
v0.1.2 延续 v0.1.0 的方式，只进行前端页面开发；涉及后端数据展示的部分统一使用 Mock 数据，通过渐进式沟通逐步完善页面细节并落地。  
本计划只锁定执行机制、技术约束和协作方式，不预先定义本版本的具体页面覆盖范围；具体页面与模块在执行过程中按问题驱动逐步确认。

## 1. 版本定位与硬约束

### 1.1 硬约束（必须满足）
1. 本版本只进行前端页面开发。
2. 涉及后端数据展示的部分统一使用临时 Mock 数据。
3. 必须遵守以下文档基线：
4. `/Users/fangyongchao/Projects/self_management/docs/v1.0/设计规范&原型策略v1.0.md`
5. `/Users/fangyongchao/Projects/self_management/docs/v1.0/项目技术方案v1.0.md`
6. 页面细节不在本计划中一次性冻结，需通过渐进式沟通逐步确认。
7. 本计划不预先锁定本版本的具体页面和模块覆盖范围，未写入本计划的页面不等于排除在本版本之外。
8. v0.1.2 的产出必须建立在当前已落地的前端工程之上，不从零重新规划登录、注册和基础壳层。

### 1.2 当前前端基线（不通过对话变更）
1. 当前已存在 `frontend/` 前端工程，技术栈固定为：Nuxt 3 + TypeScript + PrimeVue + Pinia + Tailwind。
2. 当前已存在页面：`/login`、`/register`、`/app`。
3. 当前已存在壳层：`Minimal Shell`、`Authenticated Shell`、`auth/guest` 路由守卫、主题切换、桌面折叠导航、移动端 Drawer。
4. 当前 `/app` 已存在导航和信息架构，但这些现状只作为执行基线，不自动构成本版本的固定开发清单。
5. 当前已有页面、占位页面和后续新增页面都可以进入 v0.1.2 的渐进式规划范围，是否纳入以执行过程中的确认结果为准。

## 2. 本计划锁定的内容与不锁定的内容

### 2.1 本计划锁定的内容
1. 版本目标是基于当前前端工程继续推进页面完善，而不是重新启动一套新前端。
2. 执行方式固定为“渐进式对话 + 决策清单 + 持续实现”。
3. 技术边界固定为现有 Nuxt 3 工程、现有壳层、现有导航模式和现有状态管理方式。
4. 后端依赖统一走 Mock，不以真实后端联调作为推进前提。
5. 研发 agent 必须在实现过程中持续维护页面决策记录和阶段性验收状态。

### 2.2 本计划不预先锁定的内容
1. 本版本具体会覆盖哪些页面或模块。
2. 不同页面的优先级与先后顺序。
3. 某个页面的最终信息结构、字段粒度和视觉细节。
4. Mock 数据的完整字段集合和展示深度。
5. 是否只围绕当前占位模块推进，还是在执行中转向其他页面。

### 2.3 页面选择机制（固定）
1. 每一轮开发开始前，先基于当前前端状态和用户确认结果，确定本轮要推进的页面或模块。
2. 每一轮优先选择当前价值最高、歧义最少、可独立交付的页面范围推进。
3. 一旦某个页面进入当前轮次，研发 agent 需在“计划内置决策清单”中登记该页面的决策项。
4. 某个页面暂未进入当前轮次，只表示尚未规划，不表示从本版本范围中移除。

## 3. 渐进式对话执行机制（问题驱动）

### 3.1 对话触发条件
研发 agent 在以下场景必须发起问题确认：
1. 当前实现存在两种及以上合理方案，且会影响后续代码结构。
2. 页面核心交互无法从基线文档和当前实现直接推导。
3. 继续实现会造成明显返工风险。
4. 当前轮次页面选择尚未明确，无法安全开始实现。

### 3.2 每轮对话格式（固定）
1. 每轮最多提出 1-3 个关键问题。
2. 每个问题必须给出 2-3 个可执行选项，并标注推荐项。
3. 用户确认后，agent 立即按确认项实现，不等待整批页面全部冻结。
4. 本轮确认的决策必须写入本计划中的“计划内置决策清单”。

### 3.3 推进门槛（固定）
1. 当前阻塞问题已确认即可继续下一步开发。
2. 不要求先冻结整个版本的页面清单后再开始实现。
3. 若单轮对话未获得回答，agent 可采用“最小可用默认方案”继续，但必须把该项标注为 `pending` 并在下一轮优先追问。

## 4. 实施路线（可直接执行）

| Phase | 目标 | 输出 | 对话要求 |
|---|---|---|---|
| P1 基线对齐 | 复核当前壳层、导航、已有页面和候选页面 | 明确当前轮次可选页面范围 | 无需对话 |
| P2 页面选择 | 通过对话确认当前轮次要推进的页面或模块 | 当前轮次页面清单与首批决策项 | 必要时提问 |
| P3 页面骨架与原型 | 为当前轮次页面建立稳定骨架和统一区块结构 | 页面基础结构可渲染 | 若信息结构存在分歧则提问 |
| P4 渐进实现 | 按问题驱动细化页面交互、组件组合和 Mock 数据展示 | 当前轮次页面可演示 | 持续对话（按阻塞点） |
| P5 验收与收口 | 完成冒烟检查、手工验收、决策清单沉淀 | 当前轮次可交付成果 | 无新增重大问题即可收口 |

## 5. 页面原型与实现模板

本计划不提前锁定具体页面，但为了降低后续沟通成本，先锁定页面原型模板。研发 agent 在每轮确定页面后，应先判断它属于哪种原型，再在该原型下展开细节对话。

### 5.1 Dashboard 页面
适用于总览型页面。最小要求：
1. 页面标题与摘要说明。
2. 一组核心指标卡片。
3. 至少一个趋势或状态区块。
4. 至少一个明细或近期动态区块。

### 5.2 List 页面
适用于任务、日志、记录列表等页面。最小要求：
1. 页面标题。
2. 筛选或切换区。
3. 表格或列表区。
4. 至少一种查看详情的方式。

### 5.3 Data Explorer 页面
适用于数据集浏览、分层数据预览等页面。最小要求：
1. 页面标题。
2. 数据对象切换或选择区。
3. 数据预览区。
4. 字段说明、结构说明或辅助说明区。

### 5.4 Form / Settings 页面
适用于配置或资料编辑页面。最小要求：
1. 页面标题和分区说明。
2. 表单输入区。
3. 明确的保存、取消或测试动作。
4. 完整的校验与反馈机制。

## 6. 公共约束与实现规则

### 6.1 技术与结构约束（固定）
1. 继续使用当前 Nuxt 3 工程结构，不新起第二套前端目录。
2. 继续使用当前壳层与导航模型，除非后续对话明确要求调整。
3. 新页面默认复用 PrimeVue 官方组件，Tailwind 只用于布局、间距、响应式和局部增强。
4. 不在 v0.1.2 引入新的组件体系或新的页面状态管理方案。

### 6.2 Mock 规则（固定）
1. 页面代码不得直接在 SFC 内写死业务数据常量。
2. 所有页面数据必须通过现有 service/api 抽象层接入。
3. 后端依赖部分统一由 Mock adapter 返回，返回结构遵循现有 API 类型边界。
4. 后续接真实后端时，只替换 adapter，不重写页面层调用方式。

### 6.3 设计与体验约束（固定）
1. 保持当前壳层的视觉语言连续，不做整站风格重置。
2. 页面必须兼容 Desktop 和 Mobile，两端均可用。
3. 新页面布局不得破坏现有顶栏、侧栏和内容区节奏。
4. 页面需满足基础可访问性要求：
5. 表单控件具备标签。
6. Tab 顺序可达。
7. 焦点态清晰。
8. 错误提示同时具备文本与位置反馈。

## 7. Mock 数据与接口边界

### 7.1 数据来源策略（固定）
1. 本版本不要求新增真实后端接口。
2. 若当前 service 层缺少页面所需方法，则先补齐前端 API 边界和 Mock adapter。
3. 页面只依赖“接口边界”而不依赖最终真实字段全集。

### 7.2 页面数据定义方式（固定）
1. Mock 数据结构按当前轮次页面需要逐步定义。
2. 计划中不预先冻结所有页面的数据模型。
3. 一旦某页面进入实现轮次，其数据边界应在对话确认后补入决策清单或相应实现文档。

## 8. 测试场景（执行期必须覆盖）

### 8.1 自动化冒烟
1. 未登录访问 `/app` 被重定向到 `/login`。
2. 已登录状态下可以进入当前轮次要推进的页面。
3. 当前轮次已实现页面不再停留在单一占位提示状态。
4. 移动端视口下 Drawer 可导航到当前轮次页面。
5. 页面切换不破坏现有已完成页面行为。

### 8.2 手工验收
1. 当前轮次页面都有明确的信息结构，而不是单一卡片或单行提示。
2. 页面结构与其所属原型匹配。
3. 空状态、加载状态、错误状态可读。
4. 现有壳层、主题切换和导航模式无明显回归。
