# plan_v0.1.4 修订版（前端设计渐进式执行）

## 简述
本计划用于交付给研发 agent，作为 v0.1.4 前端设计工作的执行基线。  
v0.1.4 只进行前端设计，不进行实际 API 开发，也不在计划阶段预设本版本的具体功能范围；页面、功能、信息结构和交互细节通过渐进式沟通逐步确认。  
本计划只锁定执行机制、设计约束和协作方式，不预先定义本版本必做页面或模块覆盖范围。

## 1. 版本定位与硬约束

### 1.1 硬约束（必须满足）
1. 本版本只进行前端设计。
2. 本版本不进行实际 API 开发，不以真实后端联调作为推进前提。
3. 如需辅助页面展示或交互验证，可使用临时 Mock 数据或静态示意数据，但这些数据不构成真实接口契约。
4. 必须遵守以下文档基线：
5. `/Users/fangyongchao/Projects/self_management/docs/v1.0/设计规范&原型策略v1.0.md`
6. `/Users/fangyongchao/Projects/self_management/docs/v1.0/项目技术方案v1.0.md`
7. 页面、功能和交互细节不在本计划中一次性冻结，需通过渐进式沟通逐步确认。
8. 本计划不预先锁定本版本的具体页面、模块和功能覆盖范围，未写入本计划的页面不等于排除在本版本之外。
9. v0.1.4 的产出必须建立在当前已落地的前端工程和现有设计基线之上，不重置现有壳层和前端结构。

### 1.2 当前前端基线（不通过对话变更）
1. 当前已存在 `frontend/` 前端工程，技术栈固定为：Nuxt 3 + TypeScript + PrimeVue + Pinia + Tailwind。
2. 当前已存在 `Minimal Shell`、`Authenticated Shell`、`auth/guest` 路由守卫、主题切换、桌面折叠导航和移动端 Drawer。
3. 当前已存在若干页面和占位页面，这些现状只作为设计承载基线，不自动构成本版本的固定设计清单。
4. 当前已有页面、后续新增页面和对现有页面的重构，都可以进入 v0.1.4 的渐进式设计范围，是否纳入以执行过程中的确认结果为准。

## 2. 本计划锁定的内容与不锁定的内容

### 2.1 本计划锁定的内容
1. 版本目标是基于当前前端工程继续推进页面设计，而不是重新启动一套新前端。
2. 执行方式固定为“渐进式对话 + 决策清单 + 持续收敛”。
3. 技术承载边界固定为现有 Nuxt 3 工程、现有壳层、现有导航模式和现有状态管理方式。
4. 若需要设计验证，可以在现有前端中落地静态页面、交互原型或 Mock 展示，但不扩展真实 API 开发范围。
5. 研发 agent 必须在设计推进过程中持续维护页面决策记录和阶段性验收状态。

### 2.2 本计划不预先锁定的内容
1. 本版本具体会覆盖哪些页面、模块或功能。
2. 不同页面或功能的优先级与先后顺序。
3. 某个页面的最终信息结构、字段粒度、文案层级和视觉细节。
4. 某个功能是否会在本版本中进入设计轮次。
5. Mock 或示意数据的完整字段集合和展示深度。

### 2.3 页面与功能选择机制（固定）
1. 每一轮设计开始前，先基于当前前端状态和用户确认结果，确定本轮要推进的页面或功能。
2. 每一轮优先选择当前价值最高、歧义最少、可独立收敛的一组页面或功能推进。
3. 一旦某个页面或功能进入当前轮次，研发 agent 需在“计划内置决策清单”中登记对应决策项。
4. 某个页面或功能暂未进入当前轮次，只表示尚未规划，不表示从本版本范围中移除。

## 3. 渐进式对话执行机制（问题驱动）

### 3.1 对话触发条件
研发 agent 在以下场景必须发起问题确认：
1. 当前设计存在两种及以上合理方案，且会影响后续页面结构或交互方向。
2. 页面核心交互、信息架构或视觉方向无法从基线文档和当前实现直接推导。
3. 继续推进会造成明显返工风险。
4. 当前轮次页面或功能选择尚未明确，无法安全开始设计。

### 3.2 每轮对话格式（固定）
1. 每轮最多提出 1-3 个关键问题。
2. 每个问题必须给出 2-3 个可执行选项，并标注推荐项。
3. 用户确认后，agent 立即按确认项继续推进，不等待整批页面或功能全部冻结。
4. 本轮确认的决策必须写入本计划中的“计划内置决策清单”。

### 3.3 推进门槛（固定）
1. 当前阻塞问题已确认即可继续下一步设计。
2. 不要求先冻结整个版本的页面或功能清单后再开始推进。
3. 若单轮对话未获得回答，agent 可采用“最小可用默认方案”继续，但必须把该项标注为 `pending` 并在下一轮优先追问。

## 4. 实施路线（可直接执行）

| Phase | 目标 | 输出 | 对话要求 |
|---|---|---|---|
| P1 基线对齐 | 复核当前壳层、导航、已有页面和候选设计对象 | 明确当前轮次可选页面/功能范围 | 无需对话 |
| P2 轮次选择 | 通过对话确认当前轮次要推进的页面或功能 | 当前轮次设计对象与首批决策项 | 必要时提问 |
| P3 结构设计 | 为当前轮次页面建立稳定的信息架构、区块结构和状态设计 | 页面结构草案或原型骨架 | 若结构存在分歧则提问 |
| P4 视觉与交互细化 | 细化页面层级、组件组合、交互反馈和示意数据呈现 | 当前轮次页面设计可演示 | 持续对话（按阻塞点） |
| P5 验收与收口 | 完成设计复核、决策清单沉淀和阶段性收口 | 当前轮次设计成果可交付 | 无新增重大问题即可收口 |

## 5. 页面原型与设计模板

本计划不提前锁定具体页面，但为了降低后续沟通成本，先锁定页面原型模板。研发 agent 在每轮确定页面后，应先判断它属于哪种原型，再在该原型下展开细节对话。

### 5.1 Dashboard 页面
适用于总览型页面。最小要求：
1. 页面标题与摘要说明。
2. 一组核心指标或状态卡片。
3. 至少一个趋势、概览或重点提醒区块。
4. 至少一个明细、动态或后续动作区块。

### 5.2 List 页面
适用于任务、日志、记录列表等页面。最小要求：
1. 页面标题。
2. 筛选、搜索或切换区。
3. 表格或列表主区。
4. 至少一种查看详情或进一步操作的方式。

### 5.3 Data Explorer 页面
适用于数据集浏览、层级数据预览等页面。最小要求：
1. 页面标题。
2. 数据对象切换或选择区。
3. 数据预览区。
4. 字段说明、结构说明或辅助说明区。

### 5.4 Form / Settings 页面
适用于配置或资料编辑页面。最小要求：
1. 页面标题和分区说明。
2. 表单输入区。
3. 明确的保存、取消、测试或辅助动作。
4. 完整的校验、反馈和状态提示机制。

### 5.5 Detail / Workspace 页面
适用于单对象详情、工作台或复合操作页面。最小要求：
1. 页面标题与对象上下文。
2. 主要信息区与次要信息区。
3. 至少一个操作区。
4. 清晰的状态、返回路径和辅助说明。

## 6. 公共约束与设计规则

### 6.1 技术与结构约束（固定）
1. 继续使用当前 Nuxt 3 工程结构，不新起第二套前端目录。
2. 继续使用当前壳层与导航模型，除非后续对话明确要求调整。
3. 设计验证如需落地页面，应优先复用 PrimeVue 官方组件，Tailwind 只用于布局、间距、响应式和局部增强。
4. 不在 v0.1.4 引入新的真实 API 接入层或新的后端联调依赖。

### 6.2 数据与交互占位规则（固定）
1. 本版本不要求新增真实后端接口。
2. 如需数据承载设计验证，可使用现有 Mock 机制、静态示意数据或临时占位内容。
3. 设计阶段产生的数据结构只服务于页面表达，不自动升级为后端接口契约。
4. 本计划不预先冻结所有页面的数据模型，数据粒度按当前轮次设计需要逐步定义。

### 6.3 设计与体验约束（固定）
1. 保持当前壳层的视觉语言连续，不做整站风格重置，除非用户在对话中明确要求。
2. 页面必须兼容 Desktop 和 Mobile，两端均可用。
3. 新页面布局不得破坏现有顶栏、侧栏和内容区节奏。
4. 页面需满足基础可访问性要求：
5. 表单控件具备标签。
6. Tab 顺序可达。
7. 焦点态清晰。
8. 错误提示同时具备文本与位置反馈。

## 7. 设计交付与验收标准

### 7.1 每轮设计交付物（固定）
1. 当前轮次页面或功能选择结果。
2. 当前轮次的信息架构、页面区块结构和核心交互说明。
3. 关键状态定义：空状态、加载状态、错误状态、成功反馈或无数据状态。
4. 视觉方向和组件层级说明。
5. 更新后的决策清单。

### 7.2 手工验收（执行期必须覆盖）
1. 当前轮次页面都有明确的信息结构，而不是单一卡片或单行提示。
2. 页面结构与其所属原型匹配。
3. 空状态、加载状态、错误状态和关键反馈状态可读。
4. 页面切换不破坏现有壳层、主题切换和导航模式。
5. 本轮设计成果不依赖真实 API 即可被演示和评审。
