# plan_v0.1.0 修订版（前端渐进式规划执行）

## 简述
本计划用于交付给研发 agent，作为 v0.1.0 前端开发的执行基线。  
与上一版不同，本计划不再把大量能力定义为“本版本不做”，而是定义为“本版本目标内，但本计划阶段不做一次性细化”，通过执行过程中的渐进式对话逐步确认页面细节并落地。  
对话节奏采用“按问题驱动”，每次只确认当前阻塞问题，确认后立即继续开发。

## 1. 版本定位与硬约束

### 1.1 硬约束（必须满足）
1. 本版本只进行前端页面开发。
2. 涉及后端数据展示的部分统一使用临时 Mock 数据。
3. 必须遵守以下文档基线：
4. `/Users/fangyongchao/Projects/self_management/docs/v1.0/设计规范&原型策略v1.0.md`
5. `/Users/fangyongchao/Projects/self_management/docs/v1.0/项目技术方案v1.0.md`
6. 本版本最小交付页面固定为 3 个：
7. 登录页 `/login`
8. 注册页 `/register`
9. 登录后管理页 `/app`

### 1.2 先锁定范围（不通过对话变更）
1. 技术栈：Nuxt 3 + TypeScript + PrimeVue + Pinia + Tailwind。
2. 壳层：`/login` 与 `/register` 使用 Minimal Shell，`/app` 使用 Authenticated Shell。
3. 管理页一级菜单固定 5 项：
4. 我的连接器
5. 我的同步任务
6. 我的同步日志
7. 我的数据
8. 系统设置
9. 导航模式：Desktop 支持 Expanded/Compact，Mobile 使用 Drawer。

## 2. 规划边界（修订重点）
本计划只固定“框架与执行机制”，不一次性冻结所有页面细节。  
以下内容属于 v0.1.0 目标内事项，但在本计划中不做最终定稿，交由研发 agent 在执行中通过对话确认后实现：
1. 登录页具体字段形态、提示文案、异常态呈现。
2. 注册页字段细节、校验策略、成功反馈形式。
3. 管理页欢迎区、占位模块信息结构与交互细节。
4. 组件级视觉细节、按钮层级、空状态文案。
5. Mock 数据结构的业务字段粒度（只锁定接口边界，不锁定完整字段全集）。

## 3. 渐进式对话执行机制（问题驱动）

### 3.1 对话触发条件
研发 agent 在以下场景必须发起问题确认：
1. 当前实现存在两种及以上合理方案，且会影响后续代码结构。
2. 页面核心交互无法从基线文档直接推导。
3. 继续实现会造成明显返工风险。

### 3.2 每轮对话格式（固定）
1. 每轮最多提出 1-3 个关键问题。
2. 每个问题必须给出 2-3 个可执行选项，并标注推荐项。
3. 用户确认后，agent 立即按确认项实现，不等待整页冻结。
4. 本轮确认的决策必须写入“计划内置决策清单”。

### 3.3 推进门槛（固定）
1. 当前阻塞问题已确认即可继续下一步开发。
2. 不要求登录页/注册页/管理页整页一次性冻结后再推进。

## 4. 实施路线（可直接执行）

| Phase | 目标 | 输出 | 对话要求 |
|---|---|---|---|
| P1 基础搭建 | 初始化前端工程与主题/状态/路由基础 | 可运行工程、基础 layout、middleware 框架 | 无需对话 |
| P2 壳层与路由 | 完成 Minimal + Authenticated Shell 与三路由可访问性 | `/login` `/register` `/app` 可渲染、守卫可用 | 若壳层细节冲突则提问 |
| P3 页面渐进实现 | 按问题驱动细化三个页面 | 登录/注册/管理页逐步完善 | 持续对话（按阻塞点） |
| P4 Mock 与联动 | 完成 API 抽象与 Mock 适配，打通认证流程 | 登录、登出、路由态、占位数据联动 | 遇字段歧义再提问 |
| P5 验收与沉淀 | 冒烟验证、手工验收、决策清单收敛 | 可演示版本 + 决策闭环 | 无新增重大问题即可收口 |

## 5. 公共接口与类型（先定义边界，字段可扩展）

### 5.1 API 边界（固定）
1. `POST /auth/login`
2. `POST /auth/register`
3. `POST /auth/logout`
4. `GET /auth/me`

### 5.2 类型边界（固定）
```ts
type ApiResult<T> = {
  success: boolean;
  data?: T;
  message?: string;
  code?: string;
};

type AuthUser = {
  id: string;
  displayName: string;
  role?: string;
  [key: string]: unknown;
};

type LoginPayload = {
  principal: string;   // 登录标识，具体命名可在对话确认后映射
  password: string;
  [key: string]: unknown;
};

type RegisterPayload = {
  [key: string]: unknown; // 注册字段集合通过渐进对话确认
};

type PageDecisionStatus = "pending" | "confirmed" | "implemented";
```

### 5.3 Mock 适配层规则（固定）
1. 页面代码不得直接写死数据，必须通过 service/api 抽象层访问。
2. Mock 返回结构必须遵循 `ApiResult<T>`。
3. 后续接后端时，仅替换 adapter，不重写页面层调用方式。

## 6. 三页面最小验收标准

### 6.1 登录页 `/login`
1. 支持输入、提交、错误提示、加载状态。
2. 认证成功后进入 `/app`。
3. 已登录用户访问 `/login` 自动重定向 `/app`。

### 6.2 注册页 `/register`
1. 支持字段输入、校验提示、提交反馈。
2. 注册成功后有明确成功反馈并跳转到登录流程。
3. 所有校验文案与交互反馈可读、可定位。

### 6.3 管理页 `/app`
1. 受保护路由，未登录不可访问。
2. 包含固定 5 一级菜单。
3. 包含顶栏核心区域与主内容占位区域。
4. 在桌面与移动端均可用，移动端导航为 Drawer。

## 7. 测试场景（执行期必须覆盖）

### 7.1 自动化冒烟
1. 未登录访问 `/app` 被重定向到 `/login`。
2. 登录成功后进入 `/app`。
3. 登出后再次访问 `/app` 被拦截。
4. `/login` 与 `/register` 的 guest/auth 路由守卫行为正确。
5. 主题切换状态可持久化。
6. 移动端视口下 Drawer 可开关并可导航。

### 7.2 手工验收
1. 三页面视觉与交互符合对应壳层规范。
2. 表单可键盘操作（Tab 顺序、焦点可见）。
3. 错误提示同时具备文本与位置反馈。
4. 管理页信息结构清晰，不出现无意义视觉堆叠。
