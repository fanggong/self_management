## 2. 项目概述

### 2.1 项目定位

本项目定位为一套面向登录用户的连接器接入、数据同步与数据管理系统。平台按不同 domain 提供多种可选连接器，用户可选择自己实际使用的软件作为接入入口，并通过平台提供的连接器获取数据。系统支持用户在后台页面中完成连接器配置、同步频率设置、手动同步触发、同步结果查看以及业务数据查询与维护。

项目同时具备以下三类核心能力：

1. 用户级连接器选择与配置能力
2. 用户级数据同步与调度能力
3. 用户级业务数据展示与管理能力

### 2.2 项目目标

本项目的技术建设目标如下：

1. 建立统一的连接器接入与同步框架
2. 建立标准化的数据分层体系，支持多 domain、多连接器数据汇聚与转换
3. 建立可追踪、可审计、可补跑、可重试的数据同步链路
4. 建立与业务后台页面直接对接的数据消费层
5. 形成可通过 Docker 一键部署的完整交付形态
6. 为后续接入 dbt MCP Server 预留规范化的数据建模与元数据基础

### 2.3 一期建设范围

本期建设范围包括：

- 用户登录后的后台管理页面
- 用户级连接器配置
- 用户级同步频率配置
- 用户手动触发同步
- 用户级任务记录与同步日志查询
- 用户级业务数据查询与维护
- 平台按 domain 提供可选连接器，并支持用户选择连接器完成接入
- 多连接器原始数据接入
- 基于 dbt 的标准化转换与业务层建模
- 基于 Docker Compose 的一键式部署

### 2.4 一期不包含内容

本期不包含以下内容：

- 微服务进一步拆分
- Airflow 作为主调度与编排平台
- 跨用户共享 raw 数据池
- 多地域、多活、复杂灾备架构
- 大规模数据平台化演进
- 强制落地完整 Semantic Layer

## 3. 总体技术方案

### 3.1 技术选型结论

#### 前端

- 前端框架：Vue 3
- 前端应用框架：Nuxt 3
- 前端开发语言：TypeScript
- 组件库：PrimeVue
- 状态管理：Pinia
- 数据请求方式：Nuxt useFetch / $fetch + 统一 API 封装
- 表格方案：PrimeVue DataTable
- 表单方案：PrimeVue 组件体系 + 项目内统一封装
- 主题方案：PrimeVue Styled Mode

#### 后端

- Admin API 语言：Java
- Admin API 框架：Spring Boot
- 数据同步 Worker 语言：Python
- 数据同步 Worker 框架：Celery Worker
- 周期调度：Celery Beat
- 后端总体形态：Java + Python 双服务
- API 风格：REST
- 接口文档：OpenAPI / Swagger

#### 数据与基础设施

- 主数据库：PostgreSQL
- 缓存：Redis
- 数据分层：raw + normalized + business
- ETL 实现：Python Worker 负责 Extract/Load，dbt Core 负责 Transform
- dbt 项目分层：staging / intermediate / marts
- 权限模型：JWT + RBAC + 数据归属校验
- 数据隔离模型：单库共享 + account_id 行级隔离
- 最终部署方式：Docker
- 一键部署方式：Docker Compose

### 3.2 总体架构原则

1. 前后端职责清晰分离
2. 业务接口服务与数据同步执行服务职责分离
3. 外部连接器接入与业务管理解耦
4. 原始数据、标准化数据、业务消费数据分层管理
5. 用户权限控制与数据归属控制由后端强制保障
6. 同步任务与手动触发复用同一套执行链路
7. 数据建模、文档、测试、元数据管理统一纳入 dbt 规范

## 4. 系统架构设计

### 4.1 系统组成

本系统由以下核心组成部分构成：

1. 前端管理端
2. Admin API 服务
3. 数据同步 Worker 服务
4. dbt 转换层
5. PostgreSQL 数据库
6. Redis 缓存与任务依赖组件

### 4.2 各系统职责

平台中的“连接器”用于抽象对外部软件的标准化接入能力。以 calendar 这一 domain 为例，可选连接器可以包括 Apple Calendar、Google Calendar 等。用户在具体 domain 下选择连接器并完成配置后，系统通过相应连接器实例执行后续同步。

#### 4.2.1 前端管理端

前端管理端用于承载用户登录后的后台页面，负责以下能力：

- 用户登录后的页面导航与交互
- 连接器配置页面
- 同步任务页面
- 同步日志页面
- 业务数据列表、详情、编辑与导出页面
- 前端权限展示控制

#### 4.2.2 Admin API 服务

Admin API 服务负责以下能力：

- 用户认证与鉴权
- 角色与权限控制
- 连接器配置 CRUD
- 业务数据 CRUD
- 同步任务查询
- 手动同步任务下发
- 审计日志记录
- 数据归属校验

#### 4.2.3 数据同步 Worker 服务

数据同步 Worker 服务负责以下能力：

- 调用连接器适配层以及第三方 API 或 Python SDK
- 执行用户级数据拉取任务
- 原始数据写入 raw 层
- 记录任务执行过程与结果
- 不触发 dbt 构建，dbt 转换需通过独立入口手动执行
- 执行失败重试与补跑

#### 4.2.4 dbt 转换层

dbt 转换层负责以下能力：

- 将 raw 层数据标准化至 normalized 层
- 将 normalized 层数据建模至 business 层
- 数据质量测试
- 数据文档生成
- 元数据产物生成
- 为后续 dbt MCP Server 提供规范化上下文基础

### 4.3 架构边界

#### 4.3.1 前端与后端边界

- 前端负责页面展示、交互与视图状态管理
- 后端负责所有认证、权限、数据归属和业务逻辑处理
- 前端不承担安全边界

#### 4.3.2 Admin API 与 Worker 边界

- Admin API 负责用户、权限、配置、CRUD、查询、审计
- Worker 负责连接器接入、任务执行、原始数据落库、任务状态更新

#### 4.3.3 Worker 与 dbt 边界

- Worker 负责 Extract 与 Load
- dbt 负责 raw 之后的 Transform
- dbt 不负责直接访问外部连接器或第三方软件

## 5. 前端技术方案

### 5.1 前端技术栈

- Vue 3
- Nuxt 3
- TypeScript
- PrimeVue
- Pinia

### 5.2 选型目标

前端方案需要满足以下目标：

1. 支撑以列表、表单、弹窗、详情页为主的后台页面形态
2. 提供较强的表格能力，适配后台管理场景
3. 提供清晰的路由、布局与页面中间件组织方式
4. 与后端 REST API 形成稳定配合
5. 降低前端业务代码的实现复杂度与维护成本

### 5.3 页面结构规范

列表型后台页面采用以下标准结构：

- 筛选区
- DataTable 区
- 操作区
- 详情抽屉或弹窗（按业务需要启用）

说明：

- Detail / Edit / Dashboard 等非列表型页面可按对应页型定义结构
- 页面结构规范以提升一致性为目标，不要求所有页面强制同构

一级菜单结构固定为：

- 我的连接器
- 我的同步任务
- 我的同步日志
- 我的数据
- 系统设置

### 5.4 组件使用规范

- 表格统一使用 PrimeVue DataTable
- 输入类组件统一使用 PrimeVue Input 系列组件
- 选择类组件统一使用 PrimeVue Select / MultiSelect
- 日期时间类组件统一使用 PrimeVue DatePicker
- 弹窗统一使用 PrimeVue Dialog
- 反馈组件统一使用 Toast / ConfirmDialog / Message

### 5.5 前端状态管理规范

- 全局用户态、权限态使用 Pinia
- 页面级查询条件与分页状态按页面管理
- 服务端数据请求采用统一 API 封装
- 不引入第二套状态管理体系

### 5.6 前端权限控制规范

- 登录态校验通过 Nuxt Route Middleware 实现
- 菜单权限按角色与权限点控制
- 按钮级权限通过统一权限函数封装控制
- 数据隔离由后端保证，前端只做展示控制

## 7. 数据隔离与权限模型

### 7.1 认证方式

- JWT

### 7.2 授权方式

- RBAC

### 7.3 数据隔离模型

系统采用单库共享 + account_id 行级隔离模型。

### 7.4 隔离规则

1. 所有核心业务表必须包含 account_id
2. 所有查询接口必须按 account_id 强制过滤
3. 所有配置、任务、日志、业务数据均属于具体用户或租户
4. 前端不承担任何真正的数据隔离责任
5. 后端必须同时校验操作权限与数据归属

### 7.5 高风险操作权限

以下操作必须单独授权：

- 删除
- 手动同步
- 补跑
- 重试
- 连接器配置修改

## 8. 数据分层方案

### 8.1 分层设计目标

数据分层用于解决以下问题：

1. 保留原始来源数据，支持追溯
2. 统一多连接器结构与实体定义
3. 为后台页面提供稳定、清晰、可消费的数据层
4. 避免原始数据、转换中间数据和业务消费数据混用

### 8.2 raw 层

#### 定义

raw 层为外部连接器原始数据层。

#### 特点

- 由 Python Worker 写入
- 保留原始响应或原始记录
- 保留采集时间、任务实例、domain、connector、connector instance 等来源信息
- 不直接提供给前端查询
- 不允许后台页面直接编辑

### 8.3 normalized 层

#### 定义

normalized 层为标准化数据层。

#### 特点

- 字段命名统一
- 类型统一
- 主键规则统一
- 去重、映射、实体标准化
- 统一数据粒度与语义

### 8.4 business 层

#### 定义

business 层为业务消费数据层。

#### 特点

- 直接面向后台页面消费
- 支撑列表、详情、统计、导出
- 保留业务可维护字段
- 作为业务展示与运营操作的基础层

### 8.5 幂等与覆盖规则

- 幂等键：account_id + connector_config_id + external_id
- 写入策略：upsert
- 去重范围：仅在当前 account_id 作用域内去重
- 人工覆盖标记：manual_override
- 覆盖策略：同步任务默认不覆盖人工维护字段

---

## 9. 数据分层与 dbt 项目分层对应关系

### 9.1 对应原则

数据分层用于描述数据所处的业务语义阶段；dbt 项目分层用于描述转换逻辑在 dbt 工程中的组织方式。两者不是同一概念，但存在稳定映射关系。

### 9.2 对应关系

- raw = dbt sources
- normalized = dbt staging + intermediate
- business-read = dbt marts
- business-write = 应用服务维护的数据表，不作为 dbt 重建目标表

### 9.3 说明

#### raw 与 dbt 的关系

raw 层不作为 dbt model 层存在，而是以 dbt source 形式声明，作为 dbt 转换的输入。

#### normalized 与 dbt 的关系

normalized 层通过两个阶段实现：

1. staging：完成基础标准化
2. intermediate：完成实体拼装、去重、映射与规则处理

#### business 与 dbt 的关系

- business-read 由 dbt marts 输出，用于后台页面读取
- business-write 由应用表维护，用于业务维护、人工覆盖与审计留痕

## 10. ETL 与 dbt 方案

### 10.1 ETL 实现方式

- Extract：Python Worker
- Load：Python Worker 写入 raw 层
- Transform：dbt Core

### 10.2 方案原则

1. 外部连接器接入统一由 Python Worker 完成
2. 原始数据在 raw 层落库
3. raw 之后的标准化、建模、质量测试统一由 dbt 承接
4. 不将全部 Transform 逻辑散落在 Python 脚本中

### 10.3 dbt 项目分层

dbt 项目固定采用以下分层：

- staging
- intermediate
- marts

### 10.4 dbt 目录结构

```text
models/
  staging/
    <source_system>/
  intermediate/
    <domain>/
  marts/
    <domain>/
snapshots/
macros/
tests/
docs/
```

### 10.5 dbt 命名规范

- source：src_<source_system>
- staging：stg_<name>
- intermediate：int_<name>
- 维表：dim_<name>
- 事实表：fct_<name>
- 业务宽表：mart_<name>
- 快照：snp_<name>
- exposure：exp_<name>

### 10.6 dbt source 规范

- 所有 raw 表必须先声明为 source
- 所有 source 必须有 description
- 所有 source 必须配置 freshness
- 所有关键字段必须有说明
- dbt 模型统一通过 source() 引用 raw 表

### 10.7 materialization 规范

- staging：view
- intermediate：view 或 table
- marts：incremental table 优先
- 历史追踪：snapshot

### 10.8 文档规范

- 每个 source 必须有 description
- 每个 model 必须有 description
- 每个业务消费模型关键字段必须有 description
- 必须提供 __overview__ 文档块
- 发布流程必须执行 dbt docs generate

### 10.9 meta 规范

每个 model 必填以下 meta 字段：

- owner
- domain
- layer
- contains_pii
- consumer_type
- refresh_cadence

每个 business/public model 额外必填：

- contract_version
- sla_tier
- downstream_app

### 10.10 tags 规范

统一使用以下 tags：

- layer:staging
- layer:intermediate
- layer:mart
- domain:<domain>
- cadence:hourly|daily|manual
- consumer:admin_dashboard
- published

### 10.11 数据测试规范

最少必须包含以下测试：

- 主键：not_null + unique
- 关键外键：relationships
- 关键枚举：accepted_values
- 关键业务模型：业务一致性测试

### 10.12 dbt artifacts 规范

必须保留以下 artifacts：

- manifest.json
- catalog.json
- run_results.json
- sources.json

### 10.13 dbt MCP 规范

- 研发阶段默认使用本地 dbt MCP
- dbt 项目必须保持文档、测试、元数据、血缘信息完整
- 后续根据需要补充远程 MCP 与语义层接入

## 11. 同步任务与调度方案

### 11.1 任务类型

本系统包含以下任务类型：

- 定时同步
- 手动同步
- 补跑
- 重试

### 11.2 调度粒度

调度粒度定义为：

- account_id + connector_config_id

### 11.3 调度规则

1. 每个用户可配置自己的同步频率
2. 每个用户的连接器实例单独调度
3. 同一 account_id + connector_config_id 不允许并发执行
4. 运行中的任务禁止重复手动触发
5. 单用户单连接器实例需要进行频率限制

### 11.4 任务执行链路

1. 用户配置连接器与同步频率
2. Admin API 保存配置
3. Celery Beat 生成周期调度任务
4. Worker 执行数据拉取
5. 原始数据写入 raw 层
6. 用户在独立 dbt 入口手动触发转换
7. 业务层数据供后台页面查询使用

### 11.5 任务状态记录

任务实例至少记录以下字段：

- task_id
- account_id
- connector_config_id
- trigger_type
- status
- started_at
- finished_at
- fetched_count
- written_count
- skipped_count
- error_message

### 11.6 失败处理机制

- 自动有限次重试
- 超过重试次数进入失败状态
- 支持按用户、连接器、时间范围补跑
- 支持针对失败任务实例手动重试

## 12. 连接器配置方案

### 12.1 连接器概念定义

平台使用“连接器”作为外部软件接入的统一抽象，不再直接以“数据源”作为一线产品概念。

连接器体系包含以下层级：

- domain：平台定义的业务领域，例如 calendar、crm、mail、storage
- connector：某个 domain 下的具体软件连接器，例如 Apple Calendar、Google Calendar
- connector instance：某个用户实际配置并授权后的连接器实例

平台按 domain 向用户展示可用连接器。用户选择自己正在使用的软件后，系统基于对应连接器拉取数据并进入统一的数据同步链路。

### 12.2 配置模型

每个用户可配置自己的连接器实例。连接器实例与用户或租户强绑定，用于表示某个用户在某个 connector 上的实际接入配置。

### 12.3 连接器模型

平台中的连接器采用两层产品组织方式：

- domain：业务领域，例如 calendar、crm、mail、storage
- connector：某个 domain 下的具体软件连接器，例如 Apple Calendar、Google Calendar

用户侧实际配置的是 connector instance，用于表示某个用户在某个连接器上的具体接入配置。

### 12.4 配置内容

连接器配置至少包括：

- domain
- connector 类型
- connector instance 标识
- 访问地址或账号标识
- 凭据与密钥
- 同步频率
- 启停状态
- 初始化配置项

### 12.5 连接器选择规则

- 平台先按 domain 展示可选连接器
- 用户在指定 domain 下选择自己正在使用的软件连接器
- 平台通过对应连接器获取该用户的数据
- 连接器实例与 account_id 强绑定

### 12.6 安全要求

- 连接器凭据必须密文存储
- 连接器密钥不得明文输出到日志
- 配置变更必须记录审计日志
- 凭据类敏感字段（如 access_token、refresh_token、api_key、client_secret、password）在审计日志中必须脱敏或摘要化存储，禁止明文落库

### 12.7 生效规则

- 用户修改配置后按既定规则生效
- 同步频率与启停状态由调度链路感知并执行

## 13. 审计与可观测性方案

### 13.1 审计日志

系统必须记录以下审计行为：

- 连接器配置新增
- 连接器配置修改
- 连接器配置删除
- 手动同步触发
- 补跑
- 重试
- 业务数据新增
- 业务数据编辑
- 业务数据删除

### 13.2 审计维度

审计日志至少记录：

- operator_id
- account_id
- 操作时间
- 操作对象
- 操作类型
- 修改前值（非敏感字段）
- 修改后值（非敏感字段）

补充规则：

- 敏感字段仅记录“是否变更”与脱敏后的展示值，不记录原值
- 审计查询接口默认不返回敏感字段，除非具备审计高权限且返回值仍为脱敏结果

### 13.3 任务可观测性

系统必须支持按以下维度查看同步任务：

- 用户
- 连接器实例
- 任务类型
- 执行状态
- 时间范围

### 13.4 基础监控要求

至少需要支持以下监控维度：

- 任务成功率
- 任务失败率
- 任务耗时
- 拉取条数
- 写入条数
- 连接器异常率

## 15. 最终定稿清单

### 15.1 前端

- 前端框架：Vue 3
- 前端应用框架：Nuxt 3
- 前端开发语言：TypeScript
- 组件库：PrimeVue
- 状态管理：Pinia
- 表格方案：PrimeVue DataTable
- 表单方案：PrimeVue 组件体系 + 统一封装

### 15.2 后端

- Admin API：Java + Spring Boot
- Sync Worker：Python + Celery Worker
- 周期调度：Celery Beat
- 服务形态：Java + Python 双服务
- API 风格：REST
- 接口文档：OpenAPI / Swagger

### 15.3 数据

- 主数据库：PostgreSQL
- 缓存：Redis
- 数据分层：raw + normalized + business-read/business-write
- ETL 实现：Python Worker 负责 Extract/Load；dbt Core 负责 Transform
- raw：原始采集数据层
- normalized：标准化数据层
- business-read：业务读取消费层（dbt marts 输出）
- business-write：业务维护与人工覆盖层（应用服务维护）

### 15.4 数据安全与任务

- 数据隔离：单库共享 + account_id 行级隔离
- 权限模型：JWT + RBAC + 数据归属校验
- 调度模型：按用户连接器实例动态调度
- 手动同步：复用同一套异步任务链路
- 并发控制：同一用户同一连接器实例禁止并发
- 幂等键：account_id + connector_config_id + external_id
- 人工覆盖策略：manual_override，任务默认不覆盖

### 15.5 dbt

- dbt 项目分层：staging / intermediate / marts
- dbt 对应关系：raw = sources；normalized = staging + intermediate；business-read = marts
- dbt 命名：stg_ / int_ / dim_ / fct_ / mart_ / snp_ / exp_
- dbt source：统一使用 source()
- dbt 文档：强制 description + docs generate
- dbt 元数据：强制 meta / tags
- dbt artifacts：manifest.json、catalog.json、run_results.json、sources.json
- dbt MCP：研发阶段默认本地 MCP

### 15.6 部署

- 部署方式：Docker
- 一键部署：Docker Compose
- 启动命令：docker compose up -d
- 配置注入：环境变量
- 数据持久化：Volume
- 发布方式：CI 构建镜像 + Compose 更新
