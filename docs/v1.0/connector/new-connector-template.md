# 新增连接器模板规范

## 目的
- 提供新增连接器时必须补齐的标准模板与接入清单。
- 保证任何新连接器都能以同样的结构进入前端、后端、worker、raw、dbt 和文档体系。

## 适用范围
- 适用于所有新连接器的立项、设计、实现和验收。
- 适用于新增健康域、财务域或未来其他业务域连接器。

## 规范正文

### 使用方式
- 新增连接器时，必须先复制本模板并填写完整。
- 未填写完整模板的连接器，不得进入正式编码阶段。
- 模板中的每个栏目都必须给出明确结论，不得填“待定”后直接开始实现。

### 固定接入清单
- 前端配置
  - 在前端连接器目录中增加稳定 `connector_id`、展示名称、分类和配置字段。
  - 若支持 mock 模式，必须同步补齐 mock 行为。
- admin-api 配置与校验
  - 增加连接器元数据、配置保存、状态管理和验证规则。
  - 为新增字段和错误码提供稳定契约。
- worker/内部调用
  - 实现验证、抓取或解析能力。
  - 明确任务触发方式、状态更新和失败映射。
- raw 落表
  - 明确原始表类型、原始流、去重键和原始载荷结构。
  - 保留统一血缘字段。
- dbt
  - 补齐 staging。
  - 视需要补齐 intermediate 和 marts。
  - 同步更新 `schema.yml`、tests、lineage contract。
- 测试
  - 至少覆盖成功路径、验证失败、执行失败和去重行为。
- 文档
  - 新增连接器专属文档。
  - 更新必要的总规范文档。

### 设计与实现约束
- 新连接器必须先确定接入模式，再决定任务形态和页面形态。
- 新连接器不得绕过 `admin-api` 直接与前端建立协议。
- 新连接器不得跳过 `raw` 直接写入消费层。
- 新连接器不得只补代码不补文档。
- 新连接器不得只补原始抓取不补下游数据模型规划。

## 边界与例外
- 若新连接器暂时只接入 raw 与 staging，可以先不进入 marts，但必须在模板中明确写出原因。
- 若新连接器属于强交互解析型流程，可以不支持自动调度，但必须补齐解析会话和确认规则。
- 任何例外都必须写进对应专属文档，而不是停留在实现细节里。

## 实施清单
- 复制模板并填写元信息。
- 明确配置、验证、任务和原始流。
- 明确 dbt 落点和测试清单。
- 补齐前后端、worker、raw、dbt、文档全链路。
- 提交前逐项对照固定接入清单验收。
