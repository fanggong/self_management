# Garmin Connect 连接器规范

## 目的
- 定义 `garmin-connect` 连接器的配置、验证、同步、落库和下游模型规则。
- 保证任何涉及 Garmin 数据接入的任务都遵循统一实现方式。

## 适用范围
- 适用于 Garmin Connect 连接器的前端配置、后端管理、worker 执行、raw 落库、dbt 解析和相关测试任务。

## 规范正文

### 连接器基本定义
- `connector_id` 必须固定为 `garmin-connect`。
- 连接器分类必须归入健康数据域。
- 接入模式必须定义为 `scheduled pull`。
- 该连接器必须同时支持：
  - 配置保存与连接验证
  - 定时同步
  - 手动时间窗同步

### 配置字段规范
- Garmin Connect 的最小配置字段必须为：
  - `username`
  - `password`
- `username` 必须是可用于登录源站的稳定账户标识。
- `password` 必须作为敏感字段处理，并以密文存储。
- 未通过连接验证的配置不得被视为可运行配置。

### 验证规范
- 验证必须检查：
  - `username` 非空
  - `password` 满足最小长度要求
  - 源站认证可通过
- 认证失败必须归类为凭证错误，而不是笼统网络错误。
- 网络不可达、超时或限流必须归类为连接错误并给出可重试语义。

### 调度与同步规范
- 该连接器必须支持 cron 风格的自动调度。
- 保存配置时必须明确同步计划。
- 手动同步必须以显式时间窗为输入。
- 自动同步必须基于计划时间推导同步窗口，且必须定义 lookback 策略以覆盖迟到数据。
- 自动同步窗口策略必须保持稳定，不得在未更新文档的情况下随意缩短或放大。

### 原始流规范
- Garmin Connect 原始流至少应覆盖：
  - `profile`
  - `daily_summary`
  - `body_composition`
  - `activity`
  - `sleep`
  - `heart_rate`
- 每个原始流都必须定义其写入的原始表类型：
  - 快照型数据进入快照原始表
  - 事件型数据进入事件原始表
  - 时序型数据进入时序原始表
- 同一原始流的 `source_stream` 命名必须稳定，不得按开发人员偏好改名。

### 原始落库规范
- 每次同步都必须先完成原始抓取与原始落库，再更新同步结果。
- 原始记录必须以账户、连接器实例、外部标识和业务时间为去重基础。
- 对重复记录必须显式区分：
  - 插入
  - 更新
  - 未变化/去重
- 任务统计必须能够反映抓取数、插入数、更新数和去重数。

### 下游 dbt 规范
- Garmin Connect 的每个原始流都必须映射到对应 staging 模型。
- staging 模型命名必须以 `stg_garmin_` 开头。
- 若一个原始流会拆为多个结构化模型，也必须保持同一连接器命名族。
- intermediate 模型必须承接健康域语义整理，命名必须以 `int_health_` 开头。
- 需要面向看板消费的 Garmin 数据必须继续进入 `mart_health_` 命名族。

### 任务与状态规范
- 手动和自动同步任务都必须走统一任务状态语义。
- 成功条件必须以原始同步完成为准。
- 失败任务必须保留错误消息，供任务列表和排障使用。
- 更新计划后的下一次执行时间必须与当前计划保持一致，不得依赖人工另行修正。

### 安全与限制规范
- Garmin 凭证不得回显到前端列表、任务记录或错误日志。
- 该连接器不得引入人工解析确认流程。
- 若源站限制导致部分流暂不可用，必须在专属变更中明确记录影响范围，不得静默忽略。

## 边界与例外
- 该连接器是标准定时拉取型连接器，不适用手工解析会话规范。
- 若未来新增 Garmin 原始流，必须在本文件和 dbt 文档中同步补充，不得只改代码。
- 某些 staging 模型可以从同一原始流二次展开，但不得伪造新的原始流标识。

## 实施清单
- 补配置时只允许使用 `username/password` 作为核心最小字段。
- 增加新流时先定义 `source_stream` 与原始表类型。
- 改同步窗口时先更新本文件中的窗口策略。
- 增加下游字段时同步更新 staging、intermediate、tests 和 lineage contract。
- 提交前验证任务状态、计数和错误语义可被前端消费。
