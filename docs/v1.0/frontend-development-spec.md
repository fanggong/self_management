# 前端开发规范

## 目的
- 统一前端代码组织、状态管理、接口调用和样式扩展方式。
- 保证页面实现符合架构分层，不把业务逻辑、协议细节和展示逻辑写乱。

## 适用范围
- 适用于 `frontend` 子工程的页面、布局、组件、store、service、type、middleware 和样式文件。
- 适用于新增功能、重构、联调、mock 数据补齐和 UI 修复。

## 规范正文

### 目录职责
- `pages/`
  只负责页面级编排、路由承接和局部交互组合。
- `layouts/`
  只负责页面骨架，不负责具体业务数据编排。
- `components/`
  负责可复用业务组件和通用展示组件，不直接承担路由职责。
- `services/api/`
  负责对 `admin-api` 的 HTTP 协议适配，是前端访问真实后端的唯一入口。
- `services/mock/`
  负责 mock 模式下的行为模拟，其契约必须与 `services/api/` 对齐。
- `services/<domain>/`
  负责领域级展示辅助、格式化和轻量视图映射。
- `stores/`
  负责跨页面或会话级状态，不得塞入纯展示临时态。
- `types/`
  负责领域类型、接口类型和共享结构定义。
- `middleware/`
  负责路由访问控制，不承载业务数据加载。

### 页面与组件边界
- 页面不得直接使用底层 `$fetch` 或自己拼装接口地址。
- 页面必须通过 `services/api` 或 store 间接访问后端。
- 页面只负责：
  - 页面级状态组合
  - 交互流程组织
  - 组件装配
- 组件必须优先保持单一职责，不得同时承担：
  - 路由控制
  - 接口协议处理
  - 大量跨域业务编排

### 接口调用规范
- 所有真实接口调用必须经过统一的请求封装。
- 前端必须把后端返回视为 `ApiResult` 契约，而不是页面自行猜测响应结构。
- `http` 与 `mock` 两种模式必须保持同一方法名、同一字段结构和相近错误语义。
- 任何新功能只要允许在 mock 模式下访问，就必须同步补齐 mock 实现。
- 页面不得直接保存后端原始异常对象，必须转换为用户可读消息。

### 鉴权规范
- 鉴权 token 必须使用统一 cookie 键保存。
- 所有需要鉴权的请求必须通过统一请求层注入 `Authorization` 头。
- 收到未授权响应时，必须统一清理 token 并回到登录态。
- 页面、组件和 service 不得各自维护独立的鉴权状态真相。

### 状态管理规范
- 跨页面状态必须放在 store。
- 单页面短生命周期状态应优先保留在页面或组件内部。
- store 只保存：
  - 用户身份
  - 主题/导航等全局 UI 状态
  - 跨页面共享业务态
- store 不得保存纯粹的弹窗开关、单次表单草稿或局部 hover 状态，除非这些状态必须跨组件共享。

### 表单与时间输入规范
- 表单输入必须先做前端基本校验，再发起提交。
- 与后端约定格式的日期时间字段必须使用统一格式，不得在不同页面各自发明格式。
- 上传、解析、运行等长流程表单必须有忙碌态锁定规则，防止重复提交。
- 敏感字段必须使用密码型组件或等效安全控件。

### 类型规范
- 接口类型必须集中在 `types/` 中定义，不得散落在页面和组件内部重复声明。
- 类型命名必须表达业务语义，而不是只表达 UI 位置。
- 对外接口 payload 与 response 类型必须成对维护。
- 当 mock 与 http 共用同一接口时，必须共用同一组类型定义。

### 样式规范
- 组件样式必须优先使用：
  - 主题 token
  - CSS 变量
  - Tailwind 语义类
  - PrimeVue 组件能力
- 业务页面不得大面积使用内联 style，只有动态定位、运行时尺寸或无法表达的特例可例外。
- 同类页面必须复用现有布局和样式节奏，不得重新发明新的页面壳。
- 新增品牌色、阴影、圆角或字号规范时，必须先进入主题或全局样式，而不是散落在单页中。

### 命名规范
- Vue 组件文件名必须使用 PascalCase。
- 页面文件名必须与路由语义一致。
- store 名称必须是稳定业务域名，不得使用临时任务名。
- service 方法名必须使用动词起始并体现行为，如 `list`、`getDetail`、`run`、`saveConfiguration`。
- 领域 helper 必须按功能命名，不得使用 `utils1`、`helper2` 一类模糊名称。

### 验证规范
- 页面或组件改动后，至少应完成以下验证中的适用项：
  - `npm run typecheck`
  - `npm run build`
  - mock 模式手工流程验证
  - http 模式联调验证
- 涉及接口契约变更时，必须同步验证真实接口与 mock 输出是否一致。

## 边界与例外
- 若某功能仅在服务端渲染上下文使用，也必须通过统一 service 层访问内部地址，不得绕开分层。
- 极小型一次性展示组件可以内联少量格式化逻辑，但不得内联协议处理逻辑。
- 未进入正式产品流的实验页面不得反向定义正式开发规范。

## 实施清单
- 新增页面前先确定放在 `pages`、`components` 还是 `stores`。
- 新增接口前先补齐 `types`、`services/api` 和适用的 `services/mock`。
- 新增全局状态前先判断是否真的跨页面共享。
- 新增样式前先确认是否可以复用主题 token 或现有页面壳。
- 提交前至少完成 typecheck 和一个真实用户流程验证。
