# 后端开发规范

## 目的
- 统一 `admin-api` 的接口设计、分层方式、异常模型、持久化边界和内部服务调用方式。
- 保证后端代码可维护、可测试、可作为前端与 worker 的稳定契约中心。

## 适用范围
- 适用于 `admin-api` 子工程中的 Controller、Service、Repository、Entity、内部客户端、Flyway migration 和测试代码。
- 适用于对外接口、内部接口、鉴权、事务处理、同步任务、连接器管理和 dbt 编排相关任务。

## 规范正文

### 包与分层规范
- 后端功能必须按领域分包，领域边界应稳定且可定位。
- 每个领域必须优先按以下分层组织：
  - `Controller`
    负责 HTTP 协议层、参数接收、状态码表达和结果包装。
  - `Service`
    负责业务规则、事务边界、领域编排和跨组件协作。
  - `Repository`
    负责实体查询和持久化访问。
  - `Entity/View/Request`
    负责持久化对象、出参视图和入参结构。
- Controller 不得直接编排数据库操作。
- Repository 不得承载领域决策逻辑。

### 接口设计规范
- 对外业务接口必须统一挂在 `/api/v1` 命名空间下。
- 仅供服务间使用的接口必须单独进入内部命名空间，不得与公开业务接口混用。
- JSON 接口必须统一返回 `ApiResult<T>` 语义。
- 接口成功消息、失败消息和错误码必须稳定、可读、可检索。
- 错误码必须使用大写下划线风格，如 `VALIDATION_ERROR`、`CONNECTOR_NOT_FOUND`。
- 流式接口必须显式声明流式内容类型，不得伪装成普通 JSON 接口。

### 参数校验规范
- 基础参数校验应优先使用声明式校验。
- 涉及跨字段约束、领域规则或外部依赖的校验必须放在 Service 中完成。
- 校验失败必须返回稳定错误码和可读错误消息，不得依赖底层异常直出。

### 异常规范
- 业务异常必须统一通过 `ApiException` 表达。
- 非业务异常必须由全局异常处理收口，不得向外暴露栈信息或底层实现细节。
- 同一类错误在不同接口中应复用同一错误码和相近语义。

### 鉴权规范
- 需要登录态的接口必须通过 JWT 鉴权。
- Controller 必须通过统一的认证用户映射获取当前用户，不得手写 claim 解析。
- 用户、账户、角色等上下文必须以统一认证对象向下传递。
- 内部服务调用不得复用浏览器侧 token 语义作为跨服务授权的默认方案，除非文档另有明确规定。

### 持久化规范
- 应用事务数据必须落在 `app` 相关结构中。
- 原始接入数据必须落在 `raw` 相关结构中。
- JPA 适用于事务型实体与标准仓储操作。
- `JdbcTemplate` 适用于：
  - 原始载荷 upsert
  - 批量 SQL
  - 不适合实体化的数据库操作
- 同一事务边界内不得让 JPA 与原始 SQL 互相绕开规则而形成双真相。

### 事务规范
- 事务边界必须由 Service 定义。
- Controller 不得自行控制事务。
- 长链路编排必须把“可回滚事务”和“不可回滚外部调用”区分清楚。
- 与外部系统交互时，必须先明确失败补偿或状态回写策略。

### Flyway 规范
- 数据库结构变更必须通过 Flyway migration 进入仓库。
- 历史 migration 一旦进入共享分支，不得原地修改。
- 新 migration 必须具备明确目的和最小影响范围。
- migration 必须与对应代码、文档和测试同步提交。

### 内部服务调用规范
- `admin-api` 调用 `sync-worker` 与 `dbt-runner` 时，必须通过显式客户端封装，不得在业务代码中散落原始 HTTP 调用。
- 内部客户端必须统一处理：
  - 地址拼接
  - 超时
  - 错误码映射
  - 失败消息转换
- 前端绝不能通过后门直接调用这些内部服务。

### 测试规范
- 新增或修改公开接口时，必须补足适用的 Controller 测试或 Service 测试。
- 测试必须优先覆盖：
  - 成功路径
  - 参数校验失败
  - 业务冲突
  - 内部依赖失败映射
- 测试应验证状态码、错误码和出参结构，而不是只验证方法被调用。

### 禁止事项
- 禁止 Controller 直接依赖 Repository 完成业务流程。
- 禁止在多个领域包中复制同一业务规则。
- 禁止把连接器凭证、JWT 密钥等敏感信息写入日志或响应体。
- 禁止把前端展示需要的拼装逻辑下沉到 Entity。
- 禁止以修改已有 migration 代替新增 migration。

## 边界与例外
- 对于原始接入和高性能 upsert 场景，允许使用 `JdbcTemplate` 替代 JPA，但必须保持 Service 为唯一业务入口。
- 流式 dbt 运行类接口可以不包装为普通 JSON 状态码语义，但其启动与失败约束仍应可由上层稳定处理。
- 紧急运维脚本不属于本规范主路径，但不得反向成为长期实现方式。

## 实施清单
- 新增接口前先定义路径、请求结构、返回结构和错误码。
- 写代码时先落分层，再写实现，不得边写边混层。
- 涉及数据库结构变更时先补 Flyway migration。
- 涉及内部服务调用时先封装客户端。
- 提交前补齐适用测试并验证错误语义。
