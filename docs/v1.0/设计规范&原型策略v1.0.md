# Avalon PrimeVue 模板研究与设计规范 / 原型策略

版本: v1.0  
日期: 2026-03-04

---

## 1. 文档目标

本文档基于 Avalon PrimeVue 模板公开信息、PrimeVue 官方文档与 Avalon 在线演示路由信息，对该模板进行二次研究，并沉淀为一套可直接用于产品设计、前端原型搭建和后续设计系统落地的规范。

本文档不试图复述模板的全部功能，而是回答以下问题：

1. Avalon 除了“好看”和“页面多”之外，真正值得借鉴的结构性特点是什么。
2. 如果把 Avalon 作为项目原型和设计规范的基础，应该保留什么、约束什么、改造什么。
3. 如何在 PrimeVue / Tailwind / Figma / Theme Token 的体系下，形成稳定、可扩展、可交付的设计与原型方法。

---

本结论主要基于以下来源：

- Avalon 在线演示中可被公开索引的页面与路由片段
- Avalon 公开预览图与搜索引擎收录的页面摘要

### 2.2 可信度说明

以下内容可视为高可信结论：

- 模板的官方特性列表
- PrimeVue 官方能力边界
- Avalon 演示站公开暴露的页面分类、路由层次和页面类型

- 模板背后的页面组织方法

## 3. 核心结论

Avalon 不应被理解为一个“后台首页模板”，而应被理解为一个 **完整的产品壳层系统**。它覆盖了公共页、登录认证、应用内页、内容管理、文件/消息/任务、商品与订单、帮助与异常页等多个场景，说明它的价值不在单页视觉，而在于提供了一套可扩展的应用框架。

因此，基于 Avalon 的设计策略不应是“照着某个 dashboard 画一遍”，而应是：

- 继承它的 **应用壳层模型**；
- 继承它的 **导航多状态思路**；
- 继承它的 **主题 token 化能力**；
- 继承它的 **多页面类型覆盖能力**；
- 但对视觉语言、内容优先级、交互约束和业务信息架构进行二次定义。

一句话概括：

> Avalon 最值得借鉴的，不是某张图表卡片怎么排，而是它把“企业级前端产品”拆成了一个可配置、可换肤、可切换导航模式、可横向扩展页面类型的系统。

## 4. 自主发现的模板特点

### 4.1 它不是单一后台模板，而是完整产品骨架

从公开可索引路由可以看到，Avalon 的演示站不止有 dashboard，还覆盖：

- **Dashboards**: E-Commerce、Banking、Marketing
- **Apps**: CMS（Detail / Detail-2 / List / Edit）、Chat、Files、Mail、Task List
- **E-Commerce**: Product Overview、Product List、New Product、Shopping Cart、Checkout Form、Order History、Order Summary
- **Pages**: Landing、Pricing、Contact、FAQ、Invoice、About Us、Help、CRUD、Oops、Not Found
- **Auth / System**: Login、Register、Forgot Password、New Password、Verification、Lock Screen、Access Denied、Error
- **Blocks**: Free Blocks

这说明 Avalon 的价值定位更接近于：

- SaaS 后台壳层
- 企业管理平台母板
- B 端产品原型底座
- 带部分官网与交易流程的全链路前端骨架

**设计含义**：后续如果用它做项目原型，必须先定义页面族，而不是先定义某一个 dashboard 页面。

---

### 4.2 它的核心是 App Shell，而不是内容页

Avalon 最强的部分不是“内容区”，而是整个 app shell：

- 左侧主导航
- 顶部工具栏
- 内容区网格
- 主题切换
- 菜单模式切换
- 多设备适配

换言之，它解决的是一个产品在长期迭代中的“外壳一致性”，而不是单次展示中的视觉惊艳感。

**设计含义**：原型阶段应优先定义壳层状态，而不是先做细节组件美化。

---

### 4.3 它把导航设计成“模式系统”，不是单纯 sidebar

Avalon 官方明确提供 7 种菜单模式：

- Static
- Overlay
- Slim
- Compact
- Horizontal
- Reveal
- Drawer

这说明模板对导航的理解不是“左边 240px 是否收起”，而是“导航如何以不同形态存在”。

**设计含义**：

- 导航应抽象为多状态系统，而不是 collapsed 布尔值。
- 页面布局和导航模式必须联动。
- 顶栏、主内容、面包屑、页头操作区都要感知导航状态。

---

### 4.4 它天然支持“公共站点 + 内部应用”双系统

Avalon 同时提供 Landing / Pricing / Contact / FAQ 等公共页面，以及 App、E-Commerce、User Management、Invoice 等内部业务页。这意味着它不是纯后台模板，而是支持：

- 未登录访问的品牌/营销场景
- 已登录后的业务工作台场景
- 交易与订单等半业务化流程场景

**设计含义**：

- 项目若需要同时包含官网、登录区、产品主应用，Avalon 非常适合作为统一底座。
- 设计规范必须区分 **Public Shell** 与 **Authenticated Shell**，不能混成一个页面体系。

---

### 4.5 它把主题能力建立在 Token 和 Preset 之上

Avalon 官方页说明其提供 Light / Dark 模式、8 组 surface color alternatives，以及 17 种 built-in theme colors，并支持 Aura、Lara、Nora 三套 preset。PrimeVue 官方文档进一步说明其 styled mode 基于 design tokens，preset 分为 primitive、semantic、component 三层，并可配置 darkModeSelector 与 CSS layer。

这意味着 Avalon 的换肤能力不是简单换几组 HEX 色值，而是建立在可系统扩展的 token 架构之上。

**设计含义**：

- 设计规范必须先定义 token，再定义组件视觉。
- 原型阶段就应区分 primitive / semantic / component token。
- 明暗主题切换必须在根节点层定义，而不是在组件内部零散处理。

---

### 4.6 它是“PrimeVue 组件 + Tailwind 布局/增强”路线，而不是纯 Tailwind 自造组件

Avalon 官方写明模板 built with Tailwind CSS。PrimeVue 官方又明确给出两种 Tailwind 集成路径：

1. 继续使用 styled PrimeVue 组件，在外围用 Tailwind 做布局和辅助样式；
2. 切换到 unstyled mode，通过 pass-through 在组件内部使用 Tailwind，进一步可走 Volt 这类 code ownership 模型。

Avalon 公开特征更接近第一种：**以 PrimeVue 自带组件能力为基础，再叠加 Tailwind 做布局、间距、响应式与局部样式增强**。

**设计含义**：

- 如果目标是快速高保真原型，应优先走 styled mode。
- 如果目标是中长期沉淀自有设计系统，才考虑 Volt / unstyled / pass-through 深度定制路线。

---

### 4.7 它把 Figma 文件与组件库能力拆成了两层

Avalon 官方说明：购买后可下载 Avalon 的 Figma 文件，但 PrimeVue UI components **并不包含在 Avalon Figma 文件中**，它们在 PrimeOne for Figma 中单独提供。PrimeOne Guide 也明确表示 PrimeOne 是与 Prime UI 套件实现对齐的官方 Figma 库，当前围绕 Aura Light / Aura Dark 组织。

这说明 PrimeTek 的设计交付模型其实是两层：

- **Avalon Figma**: 页面结构、布局、模板页、应用壳层
- **PrimeOne Figma UI Kit**: 与 PrimeVue 实现对齐的组件库

**设计含义**：

- 不能把 Avalon Figma 当作完整设计系统源文件。
- 正确做法是“模板页文件 + 组件库文件”并行使用。
- 原型生产时，页面壳层与组件基座应分别维护。

---

### 4.8 它适合高密度企业界面，但视觉个性默认较弱

Avalon 的公开截图普遍呈现出以下特征：

- 卡片化信息分区
- 统计摘要 + 趋势图 + 明细表组合
- 左侧菜单层级清晰
- 高信息密度但可扫描
- 强调系统秩序而非品牌叙事

其默认气质更接近“企业运营 / 管理台 / SaaS 工作台”，不适合直接作为强调品牌沉浸感、内容叙事感、情绪化表达的产品前台。

**设计含义**：

- 若项目偏 B 端或工具型产品，可直接继承 Avalon 风格。
- 若项目偏品牌化、创意化、内容沉浸式，Avalon 仅适合用作内部工作台，不适合作为统一视觉语言。

---

### 4.9 它的可扩展性强于它的独特性

PrimeVue 提供 Pass Through API，可直接访问组件内部 DOM 区段，增加样式、aria、data 属性或自定义监听器。对于模板类产品，这是极其关键的扩展机制：你不必立即 fork 组件库，也不必等待官方暴露额外 prop。

**设计含义**：

- 对于测试属性、可访问性增强、局部 DOM 微调、Tailwind 深度样式覆盖，优先使用 pass-through。
- 对于大规模 UI 改造，再评估是否进入 unstyled / Volt 模式。

---

### 4.10 它适合国际化与可访问性要求较高的产品

PrimeVue 官方给出以下能力边界：

- 组件体系具备 WCAG 2.1 AA 级别合规目标与组件级可访问性文档
- RTL 原生支持通过 `dir="rtl"` 或 CSS direction 即可启用
- RTL 当前的主要限制在 Galleria 和 Carousel

**设计含义**：

- 如果项目包含中东语言 / RTL 语言支持，Avalon 体系总体可用，但应避免将核心导航或核心业务依赖在 Carousel/Galleria 上。
- 原型与设计规范应强制要求语义化表单、键盘可达性、聚焦态、文本对比度和状态提示。

## 5. 设计规范

## 5.1 产品壳层规范

### 5.1.1 壳层分型

页面壳层必须分为三类：

1. **Public Shell**
   - 用于 Landing / Pricing / Contact / FAQ / About 等公共页面
   - 顶部主导航为主，弱化左侧导航
   - 强调品牌、转化与内容阅读

2. **Authenticated Shell**
   - 用于 Dashboard / Apps / E-Commerce / User Management / CRUD / Invoice
   - 左侧导航 + 顶栏 + 内容区三段式为主
   - 强调操作效率、信息密度与连续任务流

3. **Minimal Shell**
   - 用于 Login / Register / Forgot Password / Verification / Lock Screen / Error / Not Found
   - 减少导航干扰，突出单任务完成

### 5.1.2 壳层优先级

原型与实现时，先做壳层，再做页面。

顺序必须是：

- 定义导航状态
- 定义页头结构
- 定义内容容器和网格
- 定义全局主题与 token
- 最后再填充业务页

---

## 5.2 导航规范

### 5.2.1 导航状态模型

导航不得只定义为展开 / 收起两态，必须采用多状态模型。推荐最小状态集：

- **Expanded**: 完整左侧导航，适合复杂后台和首次进入
- **Compact**: 保留图标与选中态，节省横向空间
- **Overlay / Drawer**: 适合中小屏或需要强聚焦内容的页面

在需要更高完成度时，可进一步映射 Avalon 的：

- Static
- Overlay
- Slim
- Compact
- Horizontal
- Reveal
- Drawer

### 5.2.2 导航切换规则

- 导航按钮语义应定义为“切换导航模式”，不是“隐藏左栏”。
- 切换时必须联动以下区域：
  - 主内容区宽度
  - 页头左对齐基线
  - 面包屑宽度与截断策略
  - 二级菜单展示方式
  - 当前模块高亮方式

### 5.2.3 导航信息层级

左侧导航建议采用四层信息：

- Section Title
- Group
- Item
- Sub Item

规则：

- 一级分组数量优先控制在 5-8 个
- 单组二级入口优先控制在 3-7 个
- 长列表场景使用可折叠分组
- 当前页必须同时体现“当前页”和“所属上级分组”

---

## 5.3 页面类型规范

基于 Avalon 的页面谱系，项目原型优先拆分为以下标准页型：

### 5.3.1 Dashboard 页

结构固定为：

- 页头摘要区
- 关键指标卡片
- 趋势图 / 分布图
- 最近活动 / 异常 / 待处理事项
- 明细表格

原则：

- 仪表盘首先展示状态，其次展示趋势，最后展示明细。
- 第一屏不放复杂筛选表单。
- 每个卡片都必须有标题、当前值、对比关系或状态说明。

### 5.3.2 List 页

适用于 Product List、CMS List、User List、Files、Mail 列表等。

标准结构：

- 页头标题与主操作按钮
- 筛选 / 搜索 / 视图切换条
- 列表主体（DataTable / List / Tree / Card Grid）
- 分页 / 批量操作 / 空状态

### 5.3.3 Detail 页

适用于 CMS Detail、Product Overview、Order Summary、Invoice、Profile。

标准结构：

- 标题区与元信息
- 主信息区
- 侧边辅助信息区
- 时间线 / 关联记录 / 评论 / 附件

### 5.3.4 Edit / Form 页

适用于 New Product、CMS Edit、Checkout、Create User。

标准结构：

- 表单主区
- 分组分节
- 右侧摘要或底部固定操作区
- 表单错误反馈与保存状态反馈

### 5.3.5 Auth / Empty / Exception 页

适用于 Login、Forgot Password、Error、Not Found、Access Denied。

规则：

- 单任务页面只保留必要操作
- 避免多余导航元素
- 视觉层级突出主动作按钮

### 5.3.6 Public Marketing 页

适用于 Landing、Pricing、Contact、FAQ。

规则：

- 与后台页使用同一套 token，但版式节奏更疏朗
- 允许更强的品牌视觉与插画，但不应脱离主系统主题
- CTA 层级必须清晰

---

## 5.4 视觉系统规范

### 5.4.1 主题策略

推荐采用 PrimeVue styled mode 的 token 架构，设计 token 分为三层：

1. **Primitive Tokens**
   - 色阶
   - 灰阶
   - 圆角尺度
   - 阴影尺度
   - 间距尺度
   - 字号尺度

2. **Semantic Tokens**
   - primary
   - surface
   - text
   - muted
   - border
   - success
   - warning
   - danger
   - info

3. **Component Tokens**
   - button
   - input
   - panel
   - table
   - dialog
   - menu
   - tag
   - chart container

### 5.4.2 明暗模式规则

- 深浅主题必须共享同一语义 token 命名。
- 切换逻辑在根节点实现，避免组件内部局部判断。
- 深色模式下优先通过 surface 与 text token 调整层次，不通过大面积饱和色制造对比。
- 强调色数量控制在 1 个主色 + 1 个辅助强调色以内。

### 5.4.3 预设选择建议

基于 PrimeVue 官方对 preset 的定义：

- **Aura**: 更像 PrimeTek 的默认品牌视角，适合通用 SaaS
- **Lara**: 基于 Bootstrap 风格，适合熟悉传统业务系统的人群
- **Nora**: 明确偏企业应用气质

如果项目目标是延续 Avalon 的企业后台气质，优先从 **Nora** 或经修改的 **Aura** 出发。

### 5.4.4 色彩原则

- 主题主色只用于重点操作、高亮导航、关键指标强调
- 状态色只表示语义，不参与品牌表达
- 图表颜色必须遵循统一色板，不允许每页随意定义
- surface 层级至少定义 4 级，以支持卡片、页底、悬浮层和深浅主题转换

---

## 5.5 组件使用规范

### 5.5.1 组件使用策略

原型阶段优先使用 PrimeVue 官方组件，不建议在高保真原型阶段重新手写一套同类基础组件。

优先直接复用的组件类型：

- DataTable
- Drawer
- Dialog
- Tabs
- Stepper
- Menu / Menubar / TieredMenu / PanelMenu
- Tag / Badge / Toast / Message
- Input、Select、DatePicker、MultiSelect、TreeSelect
- Breadcrumb
- Timeline

### 5.5.2 Tailwind 的职责边界

Tailwind 主要负责：

- 页面布局
- 响应式断点
- 栅格和间距
- 局部视觉增强
- 原子级辅助样式

Tailwind 不应用于：

- 大规模替代 PrimeVue 组件语义
- 形成与 PrimeVue styled mode 完全冲突的第二套视觉系统

### 5.5.3 Pass Through 的使用边界

优先用于：

- DOM 区段 class 微调
- data-testid / aria-* 属性补充
- 局部交互钩子
- 特定组件内部布局矫正

不应用于：

- 长期堆积为隐性组件分叉
- 取代统一 token 和主题策略

---

## 5.6 信息密度规范

Avalon 的默认优势在于高密度但仍可扫描。要保留这一点，必须限制内容组织方式。

规则：

- 卡片标题始终短而明确
- 指标卡尽量 1 个主值 + 1 个对比信息 + 1 个注释
- 页面中图表数量不宜超过 3 个主图表 / 首屏
- 同一屏内表格、图表、列表不应同时超过 3 种主要信息结构
- KPI 卡片优先横向排列，避免无意义堆叠
- 视觉上优先使用留白和层级，而不是边框堆砌

---

## 5.7 响应式与多端规范

Avalon 官方强调 fully responsive 与 touch optimized，因此原型必须定义断点下的行为，而不是只做桌面图。

推荐行为：

- **Desktop**: Expanded / Compact 导航并存
- **Tablet**: Compact / Overlay 导航优先
- **Mobile**: Drawer 导航优先，顶部操作压缩为图标与关键入口

规则：

- 表格在移动端必须定义变体，不允许仅缩放桌面表格
- 页头操作在窄屏时改为 overflow menu 或二级操作区
- Dashboard 首屏在移动端转为单列卡片流
- 面包屑在窄屏可收敛为返回 + 当前页标题

---

## 5.8 可访问性与国际化规范

### 5.8.1 可访问性

最低要求：

- 所有表单控件必须有标签
- 键盘 Tab 顺序连续可达
- 明确焦点态
- Toast、Dialog、Drawer 具备正确 aria 语义
- 颜色不作为唯一状态表达方式
- 错误提示必须同时提供文本与位置反馈

### 5.8.2 RTL

- 设计规范必须预留 RTL 适配能力
- 布局中优先使用 logical properties 或可镜像结构
- 核心业务页避免依赖 Carousel / Galleria 作为关键交互容器

---

## 6. 原型策略

## 6.1 总体策略

采用 **“壳层先行、页面谱系驱动、组件复用、token 统一”** 的原型方法。

即：

1. 先定义三类 Shell
2. 再盘点页面族
3. 再定义 token
4. 再选组件策略
5. 最后做高保真页面

---

## 6.2 原型阶段建议

### Phase 1: 壳层与导航

交付内容：

- Public Shell
- Authenticated Shell
- Minimal Shell
- 导航三态或多态切换
- 顶栏、面包屑、页头、全局搜索、通知、用户菜单

目标：先把“产品骨架”做对。

### Phase 2: 页面原型母版

最少先完成以下母版：

- Dashboard
- List
- Detail
- Edit/Form
- Auth
- Exception
- Public Landing

目标：形成一组可复制的页面模板，而不是一次性页面。

### Phase 3: 业务填充

将真实业务映射到母版：

- 每个业务域选择对应页型
- 确认表格、图表、表单、详情信息的优先级
- 控制业务页面不突破壳层与 token 规范

### Phase 4: 主题与品牌化

在母版稳定后，再处理：

- 品牌色替换
- 视觉强化
- 插画 / 空状态 / 图标风格统一
- 深色模式微调

---

## 6.3 Figma 策略

推荐采用双文件体系：

1. **App Template File**
   - 壳层
   - 页面母版
   - 页面布局
   - 页面内容结构

2. **UI Kit File**
   - 基础组件
   - 交互态
   - token / variables
   - 图标与状态组件

若采用 PrimeTek 体系：

- Avalon Figma 负责模板页参考
- PrimeOne UI Kit 负责组件对齐

规则：

- 页面文件不自行复制一套基础组件
- 组件实例必须回连组件库
- 所有颜色、间距、字号通过 variables / tokens 管理

---

## 6.4 Vue / PrimeVue 原型实现策略

### 推荐路线

对于“快速做出高保真原型”的目标，推荐：

- PrimeVue **styled mode**
- Tailwind 负责布局与局部增强
- 少量 pass-through 做深度调整
- 先不进入 Volt / unstyled 模式

### 不推荐路线

在原型阶段就直接：

- 全面 unstyled
- 全量自定义 Tailwind 组件包
- 大量 fork PrimeVue 组件

原因：

- 初期速度会显著下降
- 主题一致性更难维持
- 组件维护成本会过早暴露

### 何时再升级到更深度定制

当满足以下条件时，再考虑 Volt / unstyled：

- 已有明确品牌视觉系统
- 默认 styled mode 与设计目标冲突明显
- 项目计划沉淀长期自有组件库
- 团队能维护更高复杂度的前端样式体系

---

## 6.5 页面优先级建议

如果资源有限，优先做以下页面：

1. Authenticated Shell
2. Dashboard 母版
3. List 母版
4. Detail 母版
5. Edit/Form 母版
6. Auth 页面
7. Exception 页面
8. Public Landing / Pricing / FAQ

原因：

- 这是最能验证产品是否“像一个完整系统”的最小集合
- 也是 Avalon 模板最有价值的部分

---

## 7. 推荐的设计决策

### 必须继承的部分

- App shell 三段式结构
- 多状态导航思想
- Token 化主题体系
- 明暗模式从根节点统一控制
- 高密度但卡片化的信息组织方式
- 页面母版化方法

### 必须重新定义的部分

- 品牌色与品牌情绪
- 图表色板
- 文案层级与信息优先级
- 图标使用规则
- 空状态、错误状态、成功状态文案
- 公共站点与内部系统之间的视觉边界

### 应谨慎使用的部分

- 大量模板自带示例内容
- 纯展示导向的假数据布局
- 过多小卡片堆叠
- 未经约束的多色状态标签

---

## 8. 最终落地建议

针对需要基于 Avalon 做设计规范与高保真原型的项目，建议采用以下明确路线：

### 路线结论

- **定位**: 把 Avalon 当作应用框架，不当作最终视觉答案
- **设计**: 以 Token + Shell + Page Types 为核心
- **原型**: 先 Styled Mode，再根据必要性逐步引入 Pass Through
- **Figma**: 模板页与组件库分层维护
- **实现**: PrimeVue 组件负责交互骨架，Tailwind 负责布局与增强
- **扩展**: 在需要时借助 Theme Designer、PrimeOne、Pass Through 进入更深度定制

### 最终判断

Avalon 非常适合以下类型项目：

- SaaS 后台
- 企业运营平台
- B 端管理系统
- 同时包含官网、登录区和内部应用的产品
- 需要快速产出高保真原型并逐步走向正式前端实现的项目

Avalon 不适合作为默认统一底座的场景：

- 强品牌叙事的营销站
- 以内容沉浸和视觉情绪为核心的产品前台
- 需要非常独特导航逻辑和高度实验性交互的产品

---

## 9. 可直接执行的原型行动清单

1. 确认项目只使用三类 Shell，不新增第四类临时壳层。
2. 将所有目标页面映射到 Dashboard / List / Detail / Edit / Auth / Exception / Public 七类母版。
3. 先决定导航状态模型，再设计导航按钮交互。
4. 先定义 token，再定义卡片、表格和按钮视觉。
5. 优先使用 PrimeVue styled mode 和官方组件。
6. Tailwind 仅负责布局、栅格、间距和局部增强。
7. 通过 pass-through 解决局部定制，不在原型阶段 fork 组件。
8. Figma 中分离页面模板文件与 UI Kit 文件。
9. 原型第一轮只覆盖壳层 + 7 类母版，不先做全量业务页。
10. 深色模式、响应式和空状态在第一轮原型中就纳入，而不是后补。
